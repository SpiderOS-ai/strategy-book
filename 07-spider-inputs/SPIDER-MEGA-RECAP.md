### 1.1 La these centrale

L'Europe est en train de mourir. 3 guerres perdues :

C'est le dernier tresor europeen. Ce que les Americains ne peuvent pas coder. Ce que les Chinois ne peuvent pas fabriquer.

**Et ce tresor pourrit de l'interieur.**

### 1.3 Le modele Service-as-Software

On ne vend pas un logiciel. On ne vend pas du conseil. On vend un resultat. Le logiciel FAIT le travail.

- **Ancien modele (SaaS classique)** : Acheter un outil, faire le travail soi-meme
- **Nouveau modele** : Acheter le resultat, le logiciel fait le travail
- **Modele Spider** : IA fait le travail (ratio qui evolue de 20/80 a 80/20), humain supervise, le client voit un resultat

Nathan : "je m'en fous du moyen, je veux creer un empire." Que le ratio humain/IA soit 80/20 ou 20/80, on s'en fout. Ce qui compte : la data capturee, le moat accumule, l'empire construit. Palantir a eu plus de FDE que d'ingenieurs logiciel jusqu'en 2016. Ca ne les a pas empeches de valoir $400 milliards.

### 1.4 Les 3 axes operationnels

1. **Usage interne Drakkar** : Spider dans toutes les phases de travail Drakkar (parametrage ERP, gestion de projet, conseil)
2. **Deploiement d'agents IA pour PME** : Agents specialises (relance factures, gestion campagnes, achats, reporting) deployes chez les clients

## 5. ARCHITECTURE ET PRODUIT

### 5.1 Spider Intelligence Engine -- le coeur

**AVERTISSEMENT :** Nathan previent que certains elements d'architecture sont "peut-etre inventes." Les choix de stack sont intentionnels ; certains details d'archi peuvent etre speculatifs. Nathan ajoute : "a l'ere de l'AI coding tout va beaucoup plus vite et la stack est ready de toute facon."

Le moteur d'intelligence organisationnelle est le coeur de Spider. Pipeline :

```
Sources -> Connecteurs -> Extracteur LLM -> Entity Resolver -> Claim Engine -> Analyseur -> LiveViews
```

| Composant | Role | Mecanisme |
|-----------|------|-----------|
| **Connecteurs** (7 en V1) | Ingestion depuis sources externes | Python async, adapteurs par source |
| **Extracteur LLM** | Non-structure -> claims | Haiku 4.5 (volume) -> Sonnet 4.5 (complexe) |
| **Entity Resolver** | Deduplication | Fellegi-Sunter + Jaro-Winkler. Auto-merge >=0.94 |
| **Claim Engine** | Combinaison de preuves | Dempster-Shafer. Belief/Plausibility. Conflits K>0.9 |
| **Analyseur batch** | Communautes, autorite, bus factor | networkx, scikit-learn |
| **LiveView engine** | Materialisation des 10 livrables | SurrealQL -> transform -> cache -> WebSocket |
| **ATMS simplifie** | Retractation de claims (cascade) | Recompute global en V1 (<2s pour <2000 claims) |
| **Alertes** | Anomalies (bus factor, conflits) | Pattern Datadog : rules -> check -> notify |

**Data model :**
- **Entity** : id, type (PERSON/ROLE/PROCESS/SYSTEM/COMPETENCE/...), canonical_name, aliases, tenant_id
- **Claim** : subject_id -> predicate -> object_id, belief dans [0,1], plausibility, sources[], depends_on[], status (PROPOSED/VALIDATED/REJECTED/SUPERSEDED)
- **14 predicats** : HOLDS_ROLE, REPORTS_TO, BELONGS_TO, HAS_COMPETENCE, EXECUTES_PROCESS, USES_SYSTEM, COMMUNICATES_WITH, ATTENDS_MEETING, SEQUENCE_AFTER, SIPOC_SUPPLIER, SIPOC_CUSTOMER, CONTAINS_DATA, PORTER_ACTIVITY, RISK_EXPOSURE, TRANSFORMATION_ACTION

Chaque claim en conflit detecte = opportunite de consulting. "Vos donnees disent que Pierre est CEO et directeur de production. Lequel est vrai ?" = mission de clarification organisationnelle.

**Estimation cout LLM :** Le chiffre de "~70 EUR/an pour une PME de 150 personnes" existe dans les docs mais Nathan dit : "ca ne vaut rien ce chiffre, c'est de la merde."

**La fenetre de 24 mois -- reelle pour la France ?** Nathan n'en est pas sur. Ca peut etre 24 mois pour le monde (auquel cas Spider est en retard) ou 3-5 ans pour les PME francaises (auquel cas il y a du temps). Benoit Vasseur du marche francais confirme l'inertie massive. Pas de certitude.

**Spider peut-il capturer le tacite la ou Drakkar a echoue ?** Nathan l'admet : Drakkar a echoue chez Decotec. Le tacite n'a pas ete capture. L'experience humaine est reelle mais non formalisee. Spider doit resoudre ce probleme avec l'IA -- mais ca reste a prouver. Nathan reste ouvert au challenge.

**Le human-in-the-loop -- feature ou bug ?** Si l'humain reste a 20%, c'est toujours une ESN avec de l'IA. Le marche valorisera-t-il Spider comme un SaaS (10-20x) ou comme une ESN (1-3x) ? Nathan : "Je m'en fous du moyen." L'obsession est l'empire, pas le multiple. Mais la question se posera aux investisseurs.

**Race to the bottom ?** Nathan le dit lui-meme a Benoit : l'IA compresse les couts. Si ca compresse les couts pour les agences, ca compressera aussi les couts pour Spider. Un freelance peut-il repliquer le service Spider pour Jordan a 200 EUR/mois avec les outils de 2027 ? La reponse possible : l'ontology accumulee + l'effet reseau. Mais en Phase 1, il n'y a pas d'effet reseau. C'est une course contre le temps. Nathan : "tres bonne question, je sais pas."

**Les Americains viendront-ils ?** La these dit non (ils ne voient pas le probleme). Mais OpenAI Frontier existe deja. Le pari est que le tacite europeen reste culturellement invisible pour les US. Pari, pas certitude.

**Le front-end elaborate est-il necessaire ?** Si Spider FAIT le travail pour le client, faut-il 58 ecrans ou juste une interface de validation minimaliste ? Nathan : "ca depend de quoi on parle entre validation data et workflow metier. Bonne question."

### 8.3 Hypotheses strategiques (de la these)

- La fenetre est de 24 mois (2026-2028) -- mais on n'est pas surs
- Les Americains ne viendront pas sur le tacite europeen -- pari
- L'effet reseau se declenche a ~1000 entreprises -- a prouver
- Le Knowledge Score deviendra un standard de valorisation M&A -- vision long terme
- Le modele Shogunat fonctionne dans la pratique -- non teste
