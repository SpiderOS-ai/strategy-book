7 Architectural Principles — Palantir's DNA

These seven principles, established during Gotham's development, became the architectural DNA carried forward into every subsequent Palantir platform:

#1 Ontology-Centric Modeling

All data is modeled as typed entities with properties and relationships — never as raw tables or unstructured blobs. The ontology IS the product.

#2 Security as First-Class Citizen

Access control is woven into the data model itself, not bolted on. Cell-level security, mandatory markings, and full audit trails are architectural requirements, not features.

#3 Orchestrate, Don't Replace

Palantir sits on top of existing systems, ingesting and unifying their data. It does not replace ERP/CRM/SCADA systems — it orchestrates them.

#4 Human-in-the-Loop

Automation augments human analysts; it never fully replaces them. Critical decisions require human review and approval.

#5 FDE Model (Forward Deployed Engineers)

Engineers are embedded on-site with customers to configure, adapt, and operate the platform. The deployment model IS the product strategy.

#6 Edge / Air-Gap Readiness

Every component must function in disconnected, air-gapped environments — from submarines to forward operating bases. No internet dependency.

#7 Full Provenance Tracking

Every data point is traceable to its source. Every transformation is logged. Every access is audited. The chain of custody is unbroken from raw data to analyst conclusion.

Section A2 The Ontology — Deep Technical

The Ontology is the nucleus of every Palantir product. It is not merely a metadata catalog or a schema registry — it is a live, operational data model through which all data is accessed, all actions are executed, all AI reasons, and all security is enforced. Understanding the Ontology at the implementation level is the single most important prerequisite for replicating Palantir's architecture.

6 Primitives Confirmed

The Ontology is constructed from exactly six primitive types. Everything in Palantir — every UI widget, every AI agent tool, every API endpoint, every security rule — operates on these six primitives:

# Primitive Analogous To Description Examples
1 Object Types Tables / Entity Classes The fundamental entity definitions in the business domain. Each Object Type defines a class of entities with a fixed set of typed properties Client, Order, Product, Warehouse, Shipment
2 Properties Columns / Attributes Typed attributes belonging to an Object Type. Each property has a name, a type (string, integer, date, timestamp, geohash, etc.), and metadata (searchable, sortable, display name) name, amount, date, status, geolocation
3 Link Types Foreign Keys / Relationships Directed, typed relationships between Object Types. Each link has a name, a cardinality (one-to-one, one-to-many, many-to-many), and may carry its own properties "ordered by", "contains", "assigned to", "shipped from"
4 Action Types Stored Procedures / API Endpoints Typed operations that mutate the Ontology. Actions are the ONLY way to write data through the Ontology layer. They include validation rules, side effects, and audit logging "approve order", "assign task", "escalate case", "update status"
5 Interfaces Abstract Classes / Protocols Abstract types that Object Types can implement. Interfaces define a contract (set of properties and links) that multiple Object Types share, enabling polymorphic queries and UI components Trackable (implemented by Order, Shipment, Asset), Assignable (implemented by Task, Incident)

3 Layers of the Ontology Confirmed

The six primitives are organized across three architectural layers that define the Ontology's full operational surface:

Semantic Layer 1: WHAT — The Data Model

Defines the business entities, their properties, and their relationships. This is the structural backbone: Object Types, Properties, Link Types, and Interfaces. It answers the question "What exists in the world?"

  • Object Type definitions with typed properties
  • Link Type definitions with cardinality constraints
  • Interface contracts for polymorphic access
  • Schema versioning with stable identifiers (ontologyRid)

Kinetic Layer 2: HOW — Real-Time Flows

Defines how data moves through the system and how the system acts on the world. This is the operational layer: Action Types, Functions, real-time data flows, writeback pipelines, and webhook integrations.

  • Action Types with validation rules and side effects
  • Functions (TypeScript/Python computations)
  • Writeback to source systems (ERP, CRM, databases)
  • Webhooks for event-driven integration
  • Real-time streaming flows (Flink-powered)

Dynamic Layer 3: GUARDRAILS — Security & Governance

Defines the constraints that govern all access and mutation. This layer is pervasive — it applies to every read, every write, every AI inference, every UI render.

  • RBAC (Role-Based Access Control) per object, per property, per row
  • Mandatory markings (classification labels that travel with data)
  • Full audit logging of every access and mutation
  • Provenance tracking from source to derived output
  • Real-time indexing for search and discovery

5 Critical Engineering Decisions Confirmed

The following five engineering decisions define the Ontology's behavior at the implementation level. These are the decisions a builder must understand and replicate to achieve Palantir-equivalent semantics.

Decision 1: Schema Evolution — Manual Remapping, Not Automatic

Palantir deliberately chose manual remapping over automatic schema migration. When the underlying data source changes (e.g., a column is renamed in the ERP), the Ontology does NOT automatically update. An operator must explicitly remap the Ontology property to the new source column.

Implementation Details

  • Global version counter — every schema change increments a global version, enabling clients to detect stale schemas
  • ontologyRid as stable identifier — every Object Type, Property, and Link Type has a stable, immutable resource identifier (ontologyRid) that never changes, even if the display name is modified
  • Names are aliases, not IDs — the human-readable name ("Customer Name") is a display alias. The system always references the ontologyRid internally. This decouples user-facing naming from system identity
  • OSDK auto-generates types — the Ontology SDK (OSDK) auto-generates TypeScript/Python types from the schema, providing compile-time type safety for application code. When the schema evolves, the generated types update, and broken references surface as compile errors
Why This Matters for Replication The stable-identifier-plus-alias pattern is the backbone of safe schema evolution. Without it, every rename breaks every downstream consumer. With it, the system absorbs naming changes without disruption.

Confirmed — from Palantir documentation

Decision 2: Conflict Resolution — Dual-Track Writeback System

Palantir implements a dual-track writeback system where data flows from two sources into the same Ontology objects: the automated pipeline (from source systems) and manual user edits (from analysts/operators using the UI).

Implementation Details

  • User edits are first-class citizens — they are not treated as temporary overrides or annotations. They are persistent mutations to the Ontology object
  • Default resolution: "user edits win" — when a pipeline re-ingests data that conflicts with a user edit, the user's manual change is preserved by default
  • Alternative resolution: "most recent value" — based on the source timestamp. Whichever write (pipeline or user) has the most recent timestamp wins
  • Append-only edit log — all user edits are recorded in an append-only log, providing a full history of manual changes that can be replayed, audited, or rolled back
Why This Matters for Replication Most data platforms treat the pipeline as the sole source of truth and manual edits as ephemeral. Palantir's approach recognizes that operational users have domain knowledge the pipeline lacks, and their corrections must be durable.

Confirmed — from Palantir documentation

Decision 3: Indexing — Elasticsearch-Powered "Funnel" Pipeline

Object search and discovery is powered by Elasticsearch, populated through a batch pipeline called the Funnel. This is the pipeline that makes the Ontology queryable.

Batch Funnel Stages

  1. Source Read — read the latest data from the backing dataset
  2. Diff / Changelog Generation — compare with the previous version to identify changed, added, and deleted objects
  3. Merge with User Edits — apply the edit log to resolve conflicts between pipeline data and user modifications
  4. Build Search Artifacts — construct Elasticsearch documents with all indexed properties, link references, and computed fields
  5. Atomic Cutover — swap the old index with the new index atomically, so search queries never see an inconsistent intermediate state

Streaming Mode

For streaming data sources (Flink-powered), the Funnel operates in near-real-time mode, applying incremental updates to the index without full rebuilds.

Confirmed — from Palantir documentation + engineering talks

Decision 4: Actions & Transactions — Optimistic Concurrency

All writes to the Ontology go through the Actions Service, which implements optimistic concurrency control. There are no pessimistic locks. This enables high throughput while maintaining consistency.

Transaction Flow

  1. Validation — run all validation rules defined on the Action Type (type checks, business rules, permission checks)
  2. Write — prepare the mutations (creates, updates, deletes) to be committed
  3. Commit — atomically commit all mutations. If any object has been modified since the action was initiated (optimistic concurrency check), the commit fails and the client must retry
  4. Fire Side Effects — after successful commit, trigger side effects

Two Classes of Side Effects

Class Semantics Guarantees
Transactional Writeback Part of the atomic commit. Writes to the Ontology backing store in the same transaction All-or-nothing with the commit. If writeback fails, the entire action fails
Best-Effort Side Effects Asynchronous operations triggered AFTER the commit succeeds. Typically: webhook calls, email notifications, writes to external systems At-least-once execution semantics. May be retried. Not part of the transaction boundary

Confirmed — from Palantir documentation

Decision 5: Security — PBAC + Markings-Based MAC

Palantir's security model combines Purpose-Based Access Control (PBAC) with mandatory markings (MAC — Mandatory Access Control). This is the most sophisticated element of the Ontology and the hardest to replicate.

Layer 1: Resource-Level Mandatory Markings

  • Every data element (object, property value, dataset) can carry binary, conjunctive markings
  • Markings are binary (a user either has the marking or does not) and conjunctive (all required markings must be present for access)
  • Markings follow inheritance rules — derived data inherits the union of markings from all source data
  • This prevents information from flowing "downhill" to less-classified contexts

Layer 2: Ontology-Level ACL

  • Per-object RLS (Row-Level Security) — individual objects can have access rules restricting which users/groups can see them
  • Per-property column suppression — specific properties can be hidden from specific users, even if they can see the object itself (e.g., a user can see that a Patient exists but cannot see their SSN)
  • Restricted views with dynamic row predicates — views that filter objects based on runtime-evaluated predicates (e.g., "show only orders from my region")

Layer 3: Execution-Context Controls for AI

  • Identity context — the AI agent operates under the identity of the invoking user, inheriting their exact permissions
  • Purpose context scoping — the AI's access can be further restricted to a specific purpose (e.g., "this agent is for supply chain optimization and cannot access HR data even if the user can")
  • Prevents AI from becoming a cross-domain data exfiltration channel — even if a user has access to both HR and finance data, an AI agent scoped to finance cannot surface HR data in its responses

Link-Visibility Semantics

Critical Implementation Detail Link visibility must be explicitly defined and tested. Without careful implementation, an attacker can infer the existence of classified objects through: (1) traversal results that include links to objects the user cannot see, (2) edge counts that reveal the number of hidden connections, or (3) "exists" queries that confirm the existence of a linked object without revealing its content. Palantir explicitly addresses these leakage vectors.

Confirmed — from Palantir documentation + deep research

7-Step Closed Loop Confirmed

The defining architectural pattern of Palantir is the 7-step closed loop. Unlike traditional BI tools that stop at visualization (step 6), Palantir completes the loop by writing decisions back into source systems (step 7), which in turn generates new data that flows back through the pipeline.

Key Insight "BI tools stop at step 6. Palantir closes the loop." The writeback capability (step 7) is what transforms Palantir from an analytics platform into an operating system. Decisions made in the application layer flow back into source systems as real-world actions, which in turn generate new data that re-enters the pipeline. This creates a continuous operational feedback loop.

1 The FDE Model Is MORE Human-Intensive Than Initial Research Suggested

Corrections #1 (automation rate), #7 (headcount growing), and #10 (self-service ratio) all point in the same direction: humans remain central to the delivery model.

Automation covers 30–55% of deployment work depending on complexity, not the 65–80% originally estimated. Palantir itself is hiring MORE people, not fewer, even as it deploys AI tools aggressively. And only 25–35% of customers are truly self-service after 3+ years on the platform.

Builder takeaway: Plan for a team of 8–15 deployment engineers from the start, not 3–5. The "AI does everything" staffing plan will fail.

2 Minimum Viable ACV Is Higher Than Hoped — SMBs Under €10M Revenue Are Structurally Hard to Serve

Corrections #2 (ACV floor at $300K+), #3 (TAM shrinks 250–500x), and #5 (bootcamp conversion at 22%) collectively redefine the target market. The "25M European SMBs" headline was always a fantasy; the real market is 50,000–100,000 mid-market companies.

Even with aggressive cost optimization, a deployment costs at minimum ~$200K in human time. Add margins, and the floor is ~$300K ACV. Companies with €1–10M revenue simply cannot pay this. The minimum customer profile is €10M+ revenue, likely €20M+ for comfortable economics.

Builder takeaway: Target companies with €10–100M revenue from day one. Do not attempt to serve the SMB "long tail" with a deployment-intensive model. If you want to serve smaller companies, build a fully self-service product (which is a different company).

3 AI Automation Is Real but Not a Silver Bullet — Expect 2–3 More Years Before Autonomous Deployment

Correction #4 (AI agent readiness) shows that even the best AI agents score under 30% on real-world task completion. Gartner predicts 40% of agentic AI projects will be canceled by 2027. MIT data shows 95% of AI pilots fail to reach production.

This doesn't mean AI is useless — it means AI is a productivity multiplier for human teams, not a replacement for them. Each FDE can handle 2–3x more engagements with AI assistance, but the FDE still needs to exist.

Builder takeaway: Use AI to make your deployment team 2–3x more productive, not to eliminate them. Plan for human-in-the-loop as the default until at least 2028. Build AI tools as internal force multipliers, not customer-facing autonomous agents.

4 The Consulting Market Is Transforming, Not Dying — Opportunity for a Hybrid Model

Correction #8 ("AI kills consulting") shows the consulting market is growing at 26% CAGR, not declining. The firms that are winning (Accenture, BCG, McKinsey) are the ones that pivoted fastest to AI deployment services.

This means a "software + deployment" hybrid model is not a weakness — it's a feature. Customers need help deploying, integrating, and adopting AI platforms. The services revenue from deployment is real, recurring (through ongoing support), and high-margin if delivered efficiently.

Builder takeaway: Do not be ashamed of the services component. Embrace a hybrid model where software generates 50–60% of revenue and deployment services generate 40–50%. The services are the moat — they create switching costs and deep customer relationships that pure SaaS cannot match.

5 European SMB Data Maturity Is a Chicken-and-Egg Problem

Correction #9 (data maturity) reveals the deepest structural challenge: 83.3% of European SMBs still use Excel as their primary analytical tool. Only 8.9% use any AI technology. You cannot build an ontology from data that doesn't exist in structured form.

This means the platform must CREATE data structure, not just consume it. The first 3–6 months of any deployment will be data engineering: connecting ERP systems, normalizing data formats, building the foundational data layer that the platform needs to function. This is pre-value work that customers are reluctant to pay for.

Builder takeaway: Build data ingestion and structuring as a first-class product feature, not an afterthought. The value proposition must START with "we will organize your data" before it can be "we will give you insights from your data." Consider pre-built ERP connectors for the top 10 European ERP systems as the foundation layer.

The meta-correction. Taken together, these 10 corrections tell a single story: the Palantir model is more defensible than we initially thought — and therefore harder to replicate. The moat is not just software; it's the accumulated human expertise, customer relationships, deployment methodology, and institutional knowledge built over 20 years and $3B+ in cumulative R&D spend. A startup can build a narrower, faster version targeting European mid-market — but must be honest about the human capital, time, and ACV constraints involved.

The opportunity is real but smaller, slower, and more human-intensive than Phase 1 research suggested. That's not a death sentence — it's a realistic foundation for building something that actually works.
